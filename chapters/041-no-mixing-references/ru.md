# День 41: Правило: нельзя смешивать ссылки

## Аналогия из трейдинга

Представь: один трейдер читает позицию, а другой в это время её изменяет. Первый трейдер получит неконсистентные данные. Rust предотвращает это: нельзя иметь &T и &mut T одновременно.

## Почему это отдельное правило

На первый взгляд может показаться, что это просто повторение предыдущей главы про одну `&mut`. Но здесь акцент немного другой: даже одна-единственная изменяемая ссылка конфликтует с любыми ссылками для чтения.

То есть проблема не только в двух редакторах сразу. Проблема вообще в смешивании двух режимов доступа:

1. «кто-то читает текущее состояние»
2. «кто-то меняет текущее состояние»

Если это происходит одновременно, читатель может увидеть состояние «посередине изменения».

## Нельзя: & и &mut вместе

```rust
fn main() {
    let mut portfolio = vec![50000.0, 3000.0];

    let reader = &portfolio;     // неизменяемая ссылка
    // let writer = &mut portfolio; // ОШИБКА: конфликт с reader

    println!("{:?}", reader); // используем reader
    // после этого reader больше не нужен

    let writer = &mut portfolio; // теперь ОК
    writer.push(1500.0);
    println!("{:?}", portfolio);
}
```

## NLL решает большинство проблем

```rust
fn main() {
    let mut prices = vec![48000.0, 50000.0];

    let first = &prices[0]; // &T
    println!("Первая: {}", first);
    // first больше не используется — &T заканчивается

    prices.push(52000.0); // &mut неявно — ОК!
    println!("Все: {:?}", prices);
}
```

## Правило на практике

```rust
fn read_best(prices: &[f64]) -> f64 {
    prices.iter().cloned().fold(f64::NEG_INFINITY, f64::max)
}

fn add_price(prices: &mut Vec<f64>, new_price: f64) {
    prices.push(new_price);
}

fn main() {
    let mut prices = vec![48000.0, 50000.0];

    // Читаем
    let best = read_best(&prices);
    println!("Лучшая: {}", best);

    // Изменяем (после чтения — ОК)
    add_price(&mut prices, 52000.0);
    println!("Обновлено: {:?}", prices);
}
```

## Как видеть конфликт глазами компилятора

Компилятор задаёт очень простой вопрос: «Есть ли сейчас живые ссылки, которые смотрят на эти данные?»

Если ответ да, то изменять объект уже нельзя.

```rust
fn main() {
    let mut book = vec![49_900.0, 50_000.0, 50_100.0];

    let view = &book;
    println!("Снимок книги: {:?}", view);

    // book.push(50_200.0); // конфликт: view ещё активен
}
```

Даже если тебе кажется, что чтение уже «морально закончилось», компилятор принимает решение только по фактическому времени жизни ссылки.

## Законные комбинации

Полезно не только помнить запрет, но и видеть, что разрешено:

```rust
fn main() {
    let mut prices = vec![48_000.0, 49_500.0, 50_200.0];

    let a = &prices;
    let b = &prices;
    println!("Чтение через a: {:?}", a);
    println!("Чтение через b: {:?}", b);

    let writer = &mut prices;
    writer.push(51_000.0);

    println!("После изменения: {:?}", prices);
}
```

Здесь всё корректно, потому что:

1. сначала были только чтения
2. потом эти ссылки больше не использовались
3. только после этого появился `writer`

## Временная шкала очень помогает

На практике полезно думать не типами, а временем:

1. сначала открыли чтение
2. закончили чтение
3. потом открыли редактирование

Если чтение и редактирование не пересекаются по времени, Rust обычно доволен.

```rust
fn main() {
    let mut positions = vec![1.0, 0.5, 2.0];

    {
        let snapshot = &positions;
        println!("До изменения: {:?}", snapshot);
    }

    {
        let editor = &mut positions;
        editor.push(1.2);
    }

    println!("После изменения: {:?}", positions);
}
```

## Практический пример: сначала анализ, потом обновление

```rust
fn exposure(values: &[f64]) -> f64 {
    values.iter().sum()
}

fn rebalance(values: &mut Vec<f64>, max_pos: f64) {
    for value in values.iter_mut() {
        if *value > max_pos {
            *value = max_pos;
        }
    }
}

fn main() {
    let mut portfolio = vec![0.8, 1.4, 2.1, 0.6];

    let total_before = exposure(&portfolio);
    println!("Exposure до ребалансировки: {:.2}", total_before);

    rebalance(&mut portfolio, 1.5);

    let total_after = exposure(&portfolio);
    println!("Exposure после ребалансировки: {:.2}", total_after);
    println!("Портфель: {:?}", portfolio);
}
```

Здесь очень хорошо видно здоровый порядок работы:

1. сначала читаем данные и считаем метрики
2. затем отдельно изменяем данные
3. потом снова читаем уже новое состояние

## Частые ошибки

### Ошибка 1: держать ссылку для чтения дольше, чем нужно

Если ссылка нужна только для одной печати, используй её и заверши жизнь как можно раньше.

### Ошибка 2: думать, что `println!` автоматически закрывает все заимствования навсегда

Обычно да, если это было последнее использование. Но нужно смотреть на конкретный код, а не угадывать.

### Ошибка 3: пытаться обойти правило клонированием без необходимости

Иногда проблема решается проще: достаточно сократить область жизни ссылки, а не копировать все данные.

## Что мы узнали

| Комбинация | Допустимо? |
|------------|-----------|
| &T + &T | ✓ Да |
| &mut T (одна) | ✓ Да |
| &T + &mut T | ✗ Нет |
| &mut T + &mut T | ✗ Нет |
