# День 42: Dangling References — висячие указатели

## Аналогия из трейдинга

Представь: ордер исполнен и удалён из системы, но у тебя остался указатель на него. Использование такого указателя — катастрофа. В C/C++ это называется dangling pointer. Rust делает это **невозможным**.

## Почему dangling reference так опасен

Висячая ссылка — это ссылка на данные, которых больше не существует. Самая неприятная часть в том, что такой баг может выглядеть случайным:

1. иногда код как будто работает
2. иногда выдаёт мусор
3. иногда падает не там, где ошибка была допущена

Rust выбрал жёсткий подход: лучше не скомпилировать код вообще, чем позволить такой сценарий.

## Что такое dangling reference

```rust
// Этот код НЕ скомпилируется в Rust:
// fn create_dangling() -> &String {
//     let s = String::from("BTC");
//     &s  // ОШИБКА: s уничтожается, ссылка становится висячей
// }
```

## Rust предотвращает это

```rust
// ПРАВИЛЬНО: возвращаем владение, не ссылку
fn create_ticker() -> String {
    let s = String::from("BTCUSDT");
    s // возвращаем, а не &s
}

fn main() {
    let ticker = create_ticker();
    println!("{}", ticker); // безопасно!
}
```

## Ссылка не может пережить данные

```rust
fn main() {
    let reference;

    {
        let price = 50000.0;
        // reference = &price; // ОШИБКА: price живёт меньше reference
    }

    // println!("{}", reference); // уже было бы dangling
}
```

## Правильный паттерн

```rust
fn main() {
    let price = 50000.0;  // данные живут весь блок

    let reference = &price; // ссылка не переживёт данные
    println!("Цена: {}", reference); // ОК!
}
```

## Как понимать правило на временной шкале

Ссылку можно использовать только пока живёт владелец данных.

Полезно представлять это как неравенство:

1. данные должны жить достаточно долго
2. ссылка может жить столько же или меньше
3. ссылка никогда не должна жить дольше данных

Если переписать это простыми словами: сначала должны существовать данные, и только потом на них можно безопасно ссылаться.

## Почему возврат ссылки из локальной переменной не работает

```rust
// НЕПРАВИЛЬНО
// fn make_symbol() -> &String {
//     let symbol = String::from("BTCUSDT");
//     &symbol
// }
```

Здесь проблема не в синтаксисе, а в логике жизни объекта:

1. `symbol` создаётся внутри функции
2. в конце функции `symbol` уничтожается
3. наружу пытаются вернуть ссылку на уже уничтоженное значение

Поэтому правильный подход почти всегда такой: если функция создаёт новые данные, она должна вернуть владение этими данными.

## Три безопасных способа вместо dangling

### 1. Возвращать владение

```rust
fn build_order_id() -> String {
    String::from("order-1001")
}
```

### 2. Заимствовать данные у вызывающего кода

```rust
fn first_symbol(symbols: &[String]) -> &String {
    &symbols[0]
}
```

### 3. Держать владельца выше по области видимости

```rust
fn main() {
    let market = String::from("BTCUSDT");
    let view = &market;

    println!("{}", view);
}
```

Во всех трёх случаях ссылка не переживает данные.

## Практический пример: строим отчёт безопасно

```rust
fn best_symbol(symbols: &[String]) -> &String {
    let mut best = &symbols[0];

    for symbol in symbols {
        if symbol.len() > best.len() {
            best = symbol;
        }
    }

    best
}

fn report_title(symbol: &str) -> String {
    format!("Отчёт по инструменту {}", symbol)
}

fn main() {
    let watchlist = vec![
        String::from("BTCUSDT"),
        String::from("ETHUSDT"),
        String::from("SOLUSDT"),
    ];

    let best = best_symbol(&watchlist);
    let title = report_title(best);

    println!("{}", title);
    println!("Watchlist всё ещё жив: {} элементов", watchlist.len());
}
```

Здесь:

1. `watchlist` владеет строками
2. `best_symbol` возвращает ссылку на элемент внутри `watchlist`
3. это безопасно, потому что `watchlist` жив дольше, чем `best`
4. `report_title` создаёт новую `String` и возвращает уже владение

## Частые ошибки

### Ошибка 1: считать, что ссылка продлевает жизнь объекта

Нет. Ссылка вообще ничем не владеет.

### Ошибка 2: возвращать `&String`, когда функция создаёт строку сама

Если функция создала значение внутри себя, почти всегда надо вернуть `String`, а не ссылку.

### Ошибка 3: объявить ссылку снаружи, а данные внутри блока

Это классический путь к попытке сделать ссылку длиннее жизни объекта.

## Что мы узнали

| Концепция | Описание |
|-----------|----------|
| Dangling reference | Ссылка на уничтоженные данные |
| Rust это запрещает | Ошибка компиляции, не runtime |
| Ссылка ≤ время жизни данных | Всегда |
| Решение | Возвращать владение (String, Vec...) |
