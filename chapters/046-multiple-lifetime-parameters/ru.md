# День 46: Несколько параметров времени жизни

## Аналогия из трейдинга

Сложная функция может работать с несколькими источниками данных: исторические цены (живут долго) и текущий снапшот (живёт коротко). Несколько параметров lifetime позволяют точно описать такие зависимости.

## Когда одного lifetime уже мало

Если все ссылки в сигнатуре зависят от одной и той же области жизни, одного параметра `'a` достаточно. Но иногда входные данные живут независимо:

1. конфиг стратегии существует почти всё время работы программы
2. временный снапшот рынка живёт только внутри одного блока
3. функция принимает оба значения, но возвращает ссылку только на одно из них

В такой ситуации отдельные lifetime-параметры делают контракт точным.

## Два параметра lifetime

```rust
// Возвращаем ссылку привязанную к 'a, не к 'b
fn select_price<'a, 'b>(historical: &'a f64, current: &'b f64, use_hist: bool) -> &'a f64 {
    if use_hist { historical } else { historical } // возвращаем только 'a
}

fn main() {
    let hist_price = 48000.0;
    let result;
    {
        let curr_price = 50000.0;
        result = select_price(&hist_price, &curr_price, true);
    } // curr_price уничтожена, но result привязан к hist_price — ОК
    println!("Цена: {}", result);
}
```

## Разные lifetimes для разных данных

```rust
struct PriceAnalyzer<'a, 'b> {
    history: &'a [f64],      // долгоживущие данные
    snapshot: &'b [f64],     // краткосрочные данные
}

impl<'a, 'b> PriceAnalyzer<'a, 'b> {
    fn avg_history(&self) -> f64 {
        self.history.iter().sum::<f64>() / self.history.len() as f64
    }

    fn avg_snapshot(&self) -> f64 {
        self.snapshot.iter().sum::<f64>() / self.snapshot.len() as f64
    }
}

fn main() {
    let history = vec![48000.0, 49000.0, 50000.0];
    let snapshot = vec![50500.0, 51000.0];

    let analyzer = PriceAnalyzer {
        history: &history,
        snapshot: &snapshot,
    };

    println!("Среднее история: ${:.2}", analyzer.avg_history());
    println!("Среднее снапшот: ${:.2}", analyzer.avg_snapshot());
}
```

## Когда нужны несколько lifetimes

```rust
// Одного 'a достаточно: оба входа и выход связаны
fn longer<'a>(a: &'a str, b: &'a str) -> &'a str {
    if a.len() > b.len() { a } else { b }
}

// Два нужны: входы независимы, возврат от 'a
fn prefix<'a, 'b>(s: &'a str, _prefix: &'b str) -> &'a str {
    s  // возвращаем из 'a, не из 'b
}
```

## Как читать сигнатуру `<'a, 'b>`

Полезно читать её так:

1. есть одна ссылка, связанная с `'a`
2. есть другая ссылка, связанная с `'b`
3. это независимые параметры времени жизни
4. возвращаемый тип должен явно показать, к какому именно входу он привязан

Например, в такой функции результат связан только с первым параметром:

```rust
fn choose_primary<'a, 'b>(primary: &'a str, _fallback: &'b str) -> &'a str {
    primary
}
```

Второй аргумент может жить сколько угодно мало или много, потому что в результате он не участвует.

## Почему это полезно на практике

Без нескольких lifetime-параметров сигнатура была бы либо слишком жёсткой, либо неточной. Мы бы искусственно заставляли компилятор считать, что все входы должны жить одинаково долго, хотя это не так.

То есть несколько lifetime-параметров нужны не для сложности, а наоборот, для более честного описания кода.

## Практический пример: анализ стратегии и снапшота

```rust
struct StrategyView<'a, 'b> {
    name: &'a str,
    latest_signal: &'b str,
}

impl<'a, 'b> StrategyView<'a, 'b> {
    fn strategy_name(&self) -> &'a str {
        self.name
    }

    fn signal(&self) -> &'b str {
        self.latest_signal
    }
}

fn keep_name<'a, 'b>(name: &'a str, _signal: &'b str) -> &'a str {
    name
}

fn main() {
    let strategy_name = String::from("trend_following");

    {
        let intraday_signal = String::from("BUY");

        let view = StrategyView {
            name: &strategy_name,
            latest_signal: &intraday_signal,
        };

        println!("Стратегия: {}", view.strategy_name());
        println!("Сигнал: {}", view.signal());

        let stable_name = keep_name(&strategy_name, &intraday_signal);
        println!("Стабильное имя: {}", stable_name);
    }

    println!("Долгоживущее имя стратегии: {}", strategy_name);
}
```

## Когда не нужны разные lifetime-параметры

Не стоит писать `<'a, 'b>` просто потому, что ты увидел такой синтаксис в примерах.

Если все входные ссылки и выход реально должны жить одинаково долго, одного параметра обычно достаточно.

```rust
fn longer<'a>(a: &'a str, b: &'a str) -> &'a str {
    if a.len() > b.len() { a } else { b }
}
```

То есть правило очень простое:

1. один lifetime, если зависимость общая
2. несколько lifetime, если зависимости независимы

## Частые ошибки

### Ошибка 1: добавлять много lifetime-параметров без причины

Если можно описать контракт одним параметром, так и делай.

### Ошибка 2: возвращать ссылку без явного понимания, откуда она взялась

Возвращаемая ссылка всегда должна быть логически связана с одним из входов.

### Ошибка 3: думать, что `'a` и `'b` обозначают какие-то конкретные фиксированные сроки

Нет, это просто имена параметров отношений между областями жизни.

## Что мы узнали

| Синтаксис | Описание |
|-----------|----------|
| `<'a, 'b>` | Два параметра lifetime |
| `&'a T, &'b U` | Независимые времена жизни |
| Структуры с несколькими 'a | Разные поля — разные гарантии |
| Возвращаемый тип | Должен быть привязан к одному из входов |
