# День 38: Заимствование — временный доступ к данным

## Аналогия из трейдинга

Представь, что у тебя есть торговый счёт. Счёт принадлежит тебе, но иногда ты даёшь аналитику временный доступ, чтобы он посмотрел статистику, а иногда даёшь риск-менеджеру право что-то скорректировать. Важная деталь: счёт всё равно остаётся твоим. Аналитик и риск-менеджер не становятся владельцами счёта, они только временно работают с ним.

В Rust это называется заимствованием. Владелец данных остаётся прежним, а другой код получает временный доступ через ссылку.

Это ключевая идея языка:

1. Данные имеют владельца
2. Другой код может временно смотреть на эти данные
3. Иногда другой код может временно менять эти данные
4. Но доступ строго контролируется, чтобы не возникло гонок, поломок и висячих ссылок

---

## Зачем нужно заимствование

Без borrowing функции часто были бы неудобными. Если каждая функция забирала бы владение, то после любого вызова исходная переменная становилась бы недоступной.

```rust
fn print_price(price: String) {
    println!("Цена: {}", price);
}

fn main() {
    let btc = String::from("BTC = 65000");

    print_price(btc);
    // println!("{}", btc); // ОШИБКА: владение ушло в функцию
}
```

Это не всегда то, что нам нужно. Часто функция должна только посмотреть на данные, а не забрать их себе.

Для этого и существует ссылка:

```rust
fn print_price(price: &String) {
    println!("Цена: {}", price);
}

fn main() {
    let btc = String::from("BTC = 65000");

    print_price(&btc); // передаём ссылку
    println!("Исходная строка всё ещё у нас: {}", btc);
}
```

Теперь функция получила доступ к строке, но владельцем остался `btc`.

---

## Что такое ссылка

Ссылка — это способ обратиться к данным, не забирая владение.

Есть два основных вида:

1. `&T` — неизменяемая ссылка, доступ только для чтения
2. `&mut T` — изменяемая ссылка, доступ для изменения

Пока мы разбираем общую идею borrowing. Сегодня главное понять: ссылка не владеет данными, она только смотрит на них или временно их меняет.

```rust
fn main() {
    let balance = 10_000.0;

    let balance_ref = &balance;

    println!("Баланс через владельца: {}", balance);
    println!("Баланс через ссылку: {}", balance_ref);
}
```

Здесь:

1. `balance` — владелец значения
2. `balance_ref` — ссылка на это значение
3. Данных не стало две копии
4. Владелец всё ещё один

---

## Неизменяемое заимствование

Неизменяемое заимствование позволяет читать данные, но не менять их.

```rust
fn main() {
    let ticker = String::from("BTCUSDT");

    let r1 = &ticker;
    let r2 = &ticker;

    println!("Первая ссылка: {}", r1);
    println!("Вторая ссылка: {}", r2);
    println!("Владелец тоже может читать: {}", ticker);
}
```

Это разрешено, потому что чтение безопасно. Если несколько аналитиков одновременно смотрят один и тот же отчёт, конфликтов нет.

Именно поэтому Rust разрешает много ссылок `&T` одновременно.

---

## Правила borrowing

У Rust очень чёткие правила заимствования:

1. Можно создать сколько угодно ссылок `&T`
2. Или можно создать одну ссылку `&mut T`
3. Но нельзя одновременно иметь `&T` и `&mut T` к одним и тем же данным

Это фундаментальная гарантия безопасности.

```rust
fn main() {
    let mut portfolio = vec!["BTC", "ETH", "SOL"];

    let a = &portfolio;
    let b = &portfolio;
    println!("Читаем через две ссылки: {:?} и {:?}", a, b);

    // let c = &mut portfolio; // ОШИБКА: уже есть неизменяемые ссылки
}
```

Причина простая: если одни и те же данные одновременно читаются и меняются, можно получить противоречивое состояние.

В trading-аналогии это как если один модуль прямо сейчас строит отчёт по позициям, а другой в этот же момент без координации меняет объёмы внутри этого же портфеля.

---

## Почему нельзя смешивать чтение и изменение

Посмотрим на проблему глазами логики, а не только компилятора.

```rust
fn main() {
    let mut prices = vec![50000.0, 50500.0, 51000.0];

    let view = &prices;
    println!("До изменения: {:?}", view);

    // let edit = &mut prices; // если бы это разрешалось
    // edit.push(51500.0);
    // println!("Старый view: {:?}", view);
}
```

Если бы `view` продолжал жить, а `edit` изменил `Vec`, то:

1. размер вектора мог бы поменяться
2. содержимое могло бы переехать в памяти
3. старая ссылка могла бы смотреть на уже неактуальные данные

Rust запрещает это заранее, ещё до запуска программы.

---

## Borrowing через функции

Самый частый сценарий — передать ссылку в функцию, чтобы она прочитала данные и вернула результат.

```rust
fn average(prices: &Vec<f64>) -> f64 {
    let sum: f64 = prices.iter().sum();
    sum / prices.len() as f64
}

fn main() {
    let day_prices = vec![48_000.0, 49_500.0, 50_200.0, 49_900.0];

    let avg = average(&day_prices);

    println!("Средняя цена: {:.2}", avg);
    println!("Исходные данные всё ещё доступны: {:?}", day_prices);
}
```

Функция `average` не забрала владение над вектором. Она только одолжила его на время вычисления.

Это очень важный стиль программирования в Rust:

1. если функция только читает данные, передавай ссылку
2. если функция должна изменить данные, передавай `&mut`
3. если функция должна стать новым владельцем, передавай значение целиком

---

## Почему часто лучше использовать слайсы

Технически `&Vec<f64>` работает, но в Rust часто предпочитают более общий тип `&[f64]`.

```rust
fn average(prices: &[f64]) -> f64 {
    let sum: f64 = prices.iter().sum();
    sum / prices.len() as f64
}

fn main() {
    let prices_vec = vec![48_000.0, 49_000.0, 50_000.0];
    let prices_array = [48_500.0, 49_500.0, 50_500.0];

    println!("Среднее Vec: {:.2}", average(&prices_vec));
    println!("Среднее массива: {:.2}", average(&prices_array));
}
```

`&[f64]` удобнее, потому что функция становится универсальнее: она принимает и `Vec`, и массив, и часть массива.

## Время жизни заимствования

Заимствование не вечно. Оно действует только на некоторый участок кода.

```rust
fn main() {
    let mut balance = 10_000.0;

    {
        let current = &balance;
        println!("Текущий баланс: {}", current);
    }

    balance += 500.0;
    println!("Баланс после пополнения: {}", balance);
}
```

Пока ссылка `current` жива, компилятор следит, чтобы не нарушались правила borrowing. Когда ссылка выходит из области видимости, ограничение снимается.

Проще говоря:

1. ссылка появилась
2. Rust считает, что данные временно заняты
3. ссылка исчезла
4. данные снова можно использовать без этого ограничения

---

## Non-Lexical Lifetimes: borrow живёт до последнего использования

Современный Rust достаточно умён и часто завершает заимствование не в конце блока, а после последнего использования ссылки.

```rust
fn main() {
    let mut prices = vec![50_000.0, 51_000.0];

    let first_view = &prices;
    println!("Смотрим цены: {:?}", first_view);

    // После последнего использования first_view компилятор понимает,
    // что ссылка больше не нужна
    let editor = &mut prices;
    editor.push(52_000.0);

    println!("После изменения: {:?}", prices);
}
```

Раньше можно было думать: «ссылка живёт до конца блока». Сейчас правильнее думать так: «ссылка живёт до последнего использования, если компилятор может это доказать».

---

## Типичная ошибка новичка

Очень часто ошибка выглядит так:

```rust
fn main() {
    let mut orders = vec!["BUY BTC", "SELL ETH"];

    let summary = &orders;
    println!("Ордеры: {:?}", summary);

    // orders.push("BUY SOL"); // иногда здесь будет ошибка,
    // если borrow ещё считается активным
}
```

Что здесь нужно понять:

1. проблема не в `push` как таковом
2. проблема в том, что данные ещё кем-то заимствованы
3. нужно сократить жизнь ссылки или перенести изменение позже

Часто решение очень простое:

```rust
fn main() {
    let mut orders = vec!["BUY BTC", "SELL ETH"];

    {
        let summary = &orders;
        println!("Ордеры: {:?}", summary);
    }

    orders.push("BUY SOL");
    println!("После обновления: {:?}", orders);
}
```

Мы явно ограничили жизнь ссылки блоком.

---

## Borrowing не копирует данные

Это очень важно. Когда ты передаёшь `&String` или `&Vec<f64>`, Rust не создаёт полную копию содержимого. Он передаёт только ссылку на уже существующие данные.

```rust
fn print_watchlist(list: &[String]) {
    println!("Watchlist содержит {} инструментов", list.len());
}

fn main() {
    let watchlist = vec![
        String::from("BTCUSDT"),
        String::from("ETHUSDT"),
        String::from("SOLUSDT"),
    ];

    print_watchlist(&watchlist);
    println!("Исходный вектор всё ещё наш: {:?}", watchlist);
}
```

Именно поэтому borrowing так полезен для производительности: можно передавать большие структуры без дорогого клонирования.

## Практический пример: отчёт по портфелю без передачи владения

Ниже пример, в котором несколько функций читают один и тот же набор данных, но никто не забирает владение.

```rust
fn total_value(prices: &[f64]) -> f64 {
    prices.iter().sum()
}

fn max_price(prices: &[f64]) -> f64 {
    prices.iter().cloned().fold(f64::NEG_INFINITY, f64::max)
}

fn min_price(prices: &[f64]) -> f64 {
    prices.iter().cloned().fold(f64::INFINITY, f64::min)
}

fn print_report(prices: &[f64]) {
    println!("--- Отчёт по рынку ---");
    println!("Количество значений: {}", prices.len());
    println!("Минимум: {:.2}", min_price(prices));
    println!("Максимум: {:.2}", max_price(prices));
    println!("Сумма: {:.2}", total_value(prices));
    println!("Среднее: {:.2}", total_value(prices) / prices.len() as f64);
}

fn main() {
    let prices = vec![48_000.0, 49_200.0, 50_100.0, 49_700.0, 51_000.0];

    print_report(&prices);

    println!("После отчёта цены всё ещё доступны: {:?}", prices);
}
```

Что здесь важно:

1. `prices` имеет одного владельца в `main`
2. каждая функция получает только заимствование
3. данные можно передавать в несколько функций подряд
4. после всех вызовов владелец всё ещё может использовать вектор

Это и есть нормальный повседневный стиль Rust-кода.

## Частые ошибки

### Ошибка 1: передали значение вместо ссылки

```rust
fn show(symbol: String) {
    println!("{}", symbol);
}
```

Если функция не должна забирать строку, лучше так:

```rust
fn show(symbol: &str) {
    println!("{}", symbol);
}
```

### Ошибка 2: пытаемся изменить данные, пока есть ссылки для чтения

```rust
fn main() {
    let mut prices = vec![1.0, 2.0, 3.0];
    let first = &prices;
    println!("{:?}", first);

    // prices.push(4.0); // может быть запрещено
}
```

### Ошибка 3: думаем, что ссылка продлевает жизнь значения

Ссылка не владеет данными. Она не может сделать так, чтобы данные жили дольше своего владельца. Позже это приведёт нас к теме lifetimes.

---

## Как выбирать между владением и borrowing

Полезная шпаргалка:

1. Функция только читает данные: используй `&T` или `&[T]`
2. Функция должна изменить существующие данные: используй `&mut T`
3. Функция должна сохранить данные у себя или полностью их переработать: принимай владение `T`

Пример:

```rust
fn read_only(order: &str) {
    println!("Читаем ордер: {}", order);
}

fn main() {
    let order = String::from("BUY BTC 0.25");
    read_only(&order);
    println!("Ордер всё ещё доступен: {}", order);
}
```

---

## Что мы узнали

| Идея | Смысл |
|------|-------|
| `&T` | Временный доступ только для чтения |
| Владелец остаётся прежним | Borrowing не передаёт ownership |
| Можно много `&T` | Одновременное чтение безопасно |
| Нельзя смешивать `&T` и `&mut T` | Иначе чтение и изменение конфликтуют |
| Borrow живёт ограниченное время | Обычно до последнего использования ссылки |
| Через ссылки удобно передавать большие данные | Нет лишнего клонирования |
