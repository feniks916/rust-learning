# День 43: Время жизни — сколько живёт ордер

## Аналогия из трейдинга

Каждый ордер имеет время жизни: от момента создания до исполнения или отмены. В Rust время жизни (lifetime) определяет, как долго ссылка остаётся действительной.

## Что такое lifetime простыми словами

Lifetime — это не отдельный объект в памяти и не таймер во время выполнения. Это способ для компилятора понять: «достаточно ли долго живут данные, на которые кто-то ссылается?»

То есть lifetime отвечает не на вопрос «сколько секунд живёт переменная», а на вопрос «в каком участке программы эту ссылку безопасно использовать».

## Что такое lifetime

```rust
fn main() {
    let price;                   // price объявлена

    {
        let p = 50000.0_f64;    // p создана
        price = p;               // Copy, не ссылка
    }                            // p уничтожена

    println!("{}", price);       // ОК: price — Copy от p
}
```

## Компилятор отслеживает lifetimes

```rust
fn longest_ticker<'a>(a: &'a str, b: &'a str) -> &'a str {
    if a.len() >= b.len() { a } else { b }
}

fn main() {
    let t1 = String::from("BTCUSDT");
    let result;
    {
        let t2 = String::from("ETH");
        result = longest_ticker(&t1, &t2);
        println!("Длиннее: {}", result);
    }
    // result используется только внутри блока — ОК
}
```

## Lifetime — это не runtime

Lifetime проверяется **при компиляции**. В скомпилированной программе никакой overhead'а нет.

```rust
// Явная аннотация: 'a означает "ссылка живёт не меньше 'a"
fn first_price<'a>(prices: &'a [f64]) -> &'a f64 {
    &prices[0]
}

fn main() {
    let daily = vec![48000.0, 50000.0, 52000.0];
    let first = first_price(&daily);
    println!("Первая цена: {}", first);
}
```

## Когда lifetimes вообще приходится писать руками

Во многих случаях Rust умеет вывести lifetime сам. Если функция просто принимает одну ссылку и возвращает одну ссылку, аннотация часто не нужна.

Но если:

1. ссылок несколько
2. компилятору неясно, с каким входом связан результат
3. ты описываешь структуру, которая хранит ссылки

тогда явные lifetime-аннотации становятся частью контракта функции.

## Как читать запись `<'a>`

Запись `<'a>` не означает «ровно одна конкретная длительность». Она означает: «существует некоторое время жизни `'a`, и вот как связаны входы и выходы с ним».

```rust
fn identity<'a>(value: &'a str) -> &'a str {
    value
}
```

Смысл этой сигнатуры такой:

1. функция получает ссылку, живущую как минимум `'a`
2. она возвращает ссылку, которая не может жить дольше того же `'a`
3. значит, функция не создаёт новую «долгоживущую» ссылку из воздуха

## Почему lifetime нужен в `longest_ticker`

```rust
fn longest_ticker<'a>(a: &'a str, b: &'a str) -> &'a str {
    if a.len() >= b.len() { a } else { b }
}
```

Если убрать `<'a>`, компилятор увидит две входные ссылки и один ссылочный результат, но не поймёт, к какому именно входу привязан выход.

Аннотация говорит: результат живёт не дольше общей безопасной части жизни `a` и `b`.

Проще говоря, возвращать можно только то, что гарантированно живо для обеих сторон контракта.

## Практический пример: выбираем имя стратегии

```rust
fn longer_name<'a>(left: &'a str, right: &'a str) -> &'a str {
    if left.len() >= right.len() {
        left
    } else {
        right
    }
}

fn main() {
    let primary = String::from("trend_following");

    {
        let temporary = String::from("scalp");
        let selected = longer_name(&primary, &temporary);
        println!("Выбранное имя: {}", selected);
    }

    println!("Основная стратегия всё ещё доступна: {}", primary);
}
```

Этот пример безопасен, потому что `selected` используется только там, где живы оба входа.

## Lifetime и области видимости

Очень полезно связывать lifetime не с абстрактной буквой `'a`, а с конкретными блоками `{}`.

```rust
fn main() {
    let outer = String::from("BTCUSDT");

    {
        let inner = String::from("ETH");
        let longer = longest_ticker(&outer, &inner);
        println!("Длиннее: {}", longer);
    }
}
```

Здесь `longer` нельзя вынести наружу, если он может ссылаться на `inner`, потому что `inner` умрёт в конце внутреннего блока.

## Частые ошибки

### Ошибка 1: думать, что lifetime создаёт данные

Нет, lifetime ничего не создаёт. Он только описывает отношения между уже существующими ссылками.

### Ошибка 2: путать lifetime со временем выполнения

Проверка происходит на этапе компиляции, а не в рантайме.

### Ошибка 3: пытаться возвращать ссылку на локально созданное значение

Это не проблема синтаксиса аннотаций. Это фундаментально небезопасная идея, потому что локальное значение умрёт слишком рано.

## Что мы узнали

| Концепция | Описание |
|-----------|----------|
| Lifetime | Область где ссылка действительна |
| 'a | Аннотация времени жизни |
| Проверка при компиляции | Нет runtime-cost |
| Цель | Предотвращение dangling references |
