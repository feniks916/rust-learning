# День 40: Правило одной изменяемой ссылки

## Аналогия из трейдинга

Представь двух трейдеров, одновременно изменяющих один ордер: один увеличивает объём, другой меняет цену — результат непредсказуем. Rust запрещает такой сценарий: в момент времени только один может изменять данные.

## Логика правила одной `&mut`

Здесь важно понять не синтаксис, а причину. Изменение данных — это операция, которая должна происходить под полным контролем.

Если бы Rust разрешал две изменяемые ссылки сразу, возникали бы вопросы:

1. какая правка должна считаться последней
2. что делать, если обе ссылки зависят от прежнего состояния
3. как гарантировать, что структура не будет повреждена

Для простых чисел это ещё кажется терпимым, но для `Vec`, `String`, `HashMap` и сложных структур это уже прямой путь к ошибкам.

## Почему только одна &mut

```rust
fn main() {
    let mut price = 50000.0;

    let r1 = &mut price;
    // let r2 = &mut price; // ОШИБКА: data race невозможен в Rust

    *r1 = 51000.0;
    println!("{}", price);
}
```

## Правильный паттерн: последовательное использование

```rust
fn main() {
    let mut orders = vec![100.0, 200.0, 300.0];

    {
        let first = &mut orders[0];
        *first += 50.0; // модифицируем
    } // first освобождается

    {
        let second = &mut orders[1];
        *second -= 30.0;
    }

    println!("{:?}", orders);
}
```

## Non-lexical lifetimes (NLL)

Современный Rust умнее — ссылка заканчивается в момент последнего использования:

```rust
fn main() {
    let mut balance = 10000.0;

    let r1 = &mut balance;
    *r1 += 500.0;
    // r1 больше не используется — заимствование заканчивается

    let r2 = &mut balance; // ОК! r1 уже не активна
    *r2 -= 100.0;

    println!("{}", balance);
}
```

## split_at_mut: изменяем два куска

```rust
fn main() {
    let mut prices = vec![48000.0, 49000.0, 50000.0, 51000.0];
    let (left, right) = prices.split_at_mut(2);

    left[0]  += 100.0;
    right[0] -= 100.0;

    println!("{:?}", prices);
}
```

## Как думать про эксклюзивный доступ

Полезная ментальная модель такая:

1. `&T` означает «можно открыть читальный зал»
2. `&mut T` означает «комната на ремонте, вход только одному мастеру»

Пока существует `&mut`, объект считается занятым под изменение. Поэтому Rust не разрешает параллельный доступ через другую `&mut` и даже обычную `&`.

## Последовательное изменение — нормальный путь

Очень часто новичок видит ограничение и думает, что язык мешает. На практике большая часть кода и так естественно работает последовательно.

```rust
fn main() {
    let mut pnl = vec![100.0, -40.0, 70.0];

    {
        let first = &mut pnl[0];
        *first += 10.0;
    }

    {
        let second = &mut pnl[1];
        *second -= 5.0;
    }

    {
        let third = &mut pnl[2];
        *third *= 2.0;
    }

    println!("Итоговый PnL: {:?}", pnl);
}
```

Здесь мы не боремся с компилятором. Мы просто честно описываем: сначала одна правка, потом другая, потом третья.

## Что именно даёт NLL

Non-Lexical Lifetimes делают код удобнее, потому что borrow заканчивается после последнего использования ссылки, а не обязательно в конце блока.

```rust
fn main() {
    let mut risk = 0.02;

    let current = &mut risk;
    *current += 0.005;
    println!("Промежуточный риск: {}", current);

    let next = &mut risk;
    *next = (*next).min(0.03);

    println!("Финальный риск: {}", risk);
}
```

Важно понимать: дело не в магии, а в том, что компилятор видит, что первая ссылка уже больше не нужна.

## Практический пример: пошаговая корректировка книги ордеров

```rust
fn raise_bid(level: &mut f64, delta: f64) {
    *level += delta;
}

fn lower_ask(level: &mut f64, delta: f64) {
    *level -= delta;
}

fn main() {
    let mut book = vec![49_990.0, 50_000.0, 50_010.0, 50_020.0];

    {
        let best_bid = &mut book[1];
        raise_bid(best_bid, 5.0);
    }

    {
        let best_ask = &mut book[2];
        lower_ask(best_ask, 5.0);
    }

    let (bids, asks) = book.split_at_mut(2);
    bids[0] += 2.0;
    asks[1] -= 2.0;

    println!("Обновлённая книга: {:?}", book);
}
```

## Частые ошибки

### Ошибка 1: пытаться держать две `&mut` на всякий случай

Если одна ссылка уже не нужна, дай ей закончиться раньше. Часто отдельный блок `{}` решает проблему.

### Ошибка 2: думать, что правило касается только потоков

Нет, правило защищает не только от data race между потоками, но и от логически опасных одновременных изменений в обычном коде.

### Ошибка 3: игнорировать готовые безопасные инструменты

Для массивов и срезов уже есть `split_at_mut`, который позволяет честно показать компилятору, что части не пересекаются.

## Что мы узнали

| Правило | Причина |
|---------|---------|
| Одна &mut в момент времени | Предотвращение data race |
| NLL: ссылка живёт до последнего использования | Гибкость без ущерба безопасности |
| split_at_mut | Безопасное разбиение на изменяемые части |
| Последовательность через блоки {} | Классический паттерн |
